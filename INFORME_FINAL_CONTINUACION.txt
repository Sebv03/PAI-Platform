9.1. IMPLEMENTACIÓN DE LA SOLUCIÓN

La plataforma PAI (Plataforma Académica Inteligente) ha sido desarrollada como una solución completa y funcional para la gestión académica preuniversitaria, específicamente enfocada en la preparación de estudiantes de 1° a 4° medio para la Prueba de Acceso a la Educación Superior (PAES) en Chile.

9.1.1. Componentes Implementados

Frontend (React + Vite):
- Interfaz de usuario moderna y responsiva desarrollada con React 18
- Sistema de routing con React Router DOM para navegación fluida
- Gestión de estado global con Zustand para autenticación y datos del usuario
- Diseño responsivo con CSS puro, optimizado para diferentes dispositivos
- Componentes modulares reutilizables:
  * Dashboard específico por rol (Administrador, Profesor, Estudiante)
  * Formularios de creación de cursos y tareas
  * Visualización de predicciones de riesgo académico
  * Sistema de búsqueda de estudiantes con predicciones basadas en perfil
  * Panel de foro/comunicados para cada curso
  * Tabla de estudiantes inscritos con información detallada

Backend (FastAPI):
- API REST completa con FastAPI, documentación automática con Swagger/OpenAPI
- Sistema de autenticación robusto con JWT (JSON Web Tokens)
- Autorización basada en roles (Administrador, Docente, Estudiante)
- Validación de datos con Pydantic en todos los endpoints
- CRUD completo para todas las entidades del sistema:
  * Usuarios (creación, actualización, búsqueda)
  * Cursos (creación, actualización, listado, búsqueda)
  * Tareas (creación, actualización, eliminación)
  * Inscripciones (inscripción, desinscripción, consulta)
  * Entregas (subida de archivos PDF, calificación)
  * Comunicados y comentarios (foro por curso)
  * Perfiles de estudiantes (cuestionario de registro)
- Integración con microservicio ML mediante HTTP

Base de Datos (PostgreSQL):
- Modelo relacional completo con 8 tablas principales
- Relaciones bien definidas con integridad referencial
- Índices optimizados para consultas frecuentes
- Campos especializados para clasificación PAES (subject, paes_topic)
- Sistema de perfiles de estudiantes con 8 variables psicológicas/académicas

Microservicio ML (FastAPI):
- Servicio independiente para predicción de riesgo académico
- Feature Engineering avanzado con 12 features:
  * Features transaccionales: tasa de retraso, tasa de no entrega, promedio de notas, variabilidad
  * Features de perfil: motivación, tiempo disponible, horas de sueño, horas de estudio, disfrute, tranquilidad, presión académica, género
- Modelo Random Forest entrenado con scikit-learn
- Endpoints REST para entrenamiento y predicción (individual y batch)
- Predicción proactiva basada solo en perfil (antes de tener datos transaccionales)

9.1.2. Funcionalidades por Rol

Rol: Administrador
- Visualización de todos los cursos de la plataforma
- Capacidad de crear cursos de prueba y asignar estudiantes automáticamente
- Acceso a predicciones de riesgo académico de todos los cursos

- Visualización detallada de features y métricas de confianza de las predicciones

Rol: Profesor/Docente

11.1. PLAN DE DESPLIEGUE

11.1.1. Estrategia de Despliegue

La plataforma PAI utiliza una estrategia de despliegue basada en contenedores Docker para facilitar la implementación en diferentes entornos:

Entornos Definidos:
- **Desarrollo (Local)**: Servicios ejecutándose directamente en máquinas de desarrollo
- **Producción**: Servicios containerizados con Docker Compose en servidor VPS/Cloud

11.1.2. Pipeline CI/CD

Configuración Actual:
- Control de versiones con Git
- Repositorio centralizado en GitHub
- Despliegue manual mediante scripts y Docker Compose

Recomendaciones para CI/CD Automático:
1. GitHub Actions para:
   - Tests automáticos en cada push
   - Build de imágenes Docker
   - Deploy automático a staging
   - Deploy manual a producción con aprobación

2. Flujo Propuesto:
```
Código → Git Push → GitHub Actions → Tests → Build Docker → Deploy Staging → Aprobación → Deploy Producción
```

11.1.3. Proceso de Despliegue Manual (Actual)

Pasos para Despliegue:
1. Preparar servidor (Ubuntu 20.04+)
2. Instalar Docker y Docker Compose
3. Clonar repositorio
4. Configurar variables de entorno (.env)
5. Construir imágenes: `docker-compose -f docker-compose.prod.yml build`
6. Levantar servicios: `docker-compose -f docker-compose.prod.yml up -d`
7. Ejecutar migraciones: Scripts de inicialización
8. Verificar servicios: Health checks

Tiempo Estimado: 30-60 minutos

12.1. ANÁLISIS DE RIESGOS DE SEGURIDAD

12.1.1. Identificación de Riesgos

Riesgos Críticos:
1. **Acceso no autorizado a datos de estudiantes**
   - Probabilidad: Media
   - Impacto: Alto
   - Mitigación: Autenticación JWT robusta, autorización por roles, contraseñas hasheadas

2. **Inyección SQL**
   - Probabilidad: Baja
   - Impacto: Crítico
   - Mitigación: Uso de ORM (SQLAlchemy) con queries parametrizadas, validación Pydantic

3. **Cross-Site Scripting (XSS)**
   - Probabilidad: Media
   - Impacto: Medio
   - Mitigación: Sanitización de inputs, React escapa automáticamente el contenido

4. **Exposición de información sensible**
   - Probabilidad: Media
   - Impacto: Alto
   - Mitigación: Variables de entorno para secretos, tokens JWT con expiración

5. **Ataques de fuerza bruta**
   - Probabilidad: Media
   - Impacto: Medio
   - Mitigación: Rate limiting (a implementar), contraseñas complejas requeridas

12.1.2. OWASP Top 10 - Análisis

A01:2021 – Broken Access Control
- ✅ Implementado: Autorización basada en roles en todos los endpoints
- ✅ Verificación: Cada endpoint valida el rol del usuario antes de procesar

A02:2021 – Cryptographic Failures
- ✅ Implementado: Contraseñas hasheadas con bcrypt (salt automático)
- ✅ Implementado: JWT firmados con algoritmo HS256 y secret key

A03:2021 – Injection
- ✅ Implementado: SQLAlchemy ORM previene inyección SQL
- ✅ Implementado: Validación Pydantic en todos los inputs

A04:2021 – Insecure Design
- ✅ Arquitectura de seguridad por diseño
- ✅ Separación de responsabilidades
- ✅ Principio de menor privilegio en roles

A05:2021 – Security Misconfiguration
- ⚠️ Mejora recomendada: Headers de seguridad HTTP (CSP, HSTS)
- ⚠️ Mejora recomendada: Deshabilitar información de versión en headers

A06:2021 – Vulnerable Components
- ✅ Dependencias actualizadas regularmente
- ✅ Uso de versiones estables y mantenidas

A07:2021 – Authentication Failures
- ✅ JWT con expiración configurable
- ✅ Contraseñas hasheadas con bcrypt
- ⚠️ Mejora recomendada: Rate limiting en endpoints de autenticación

A08:2021 – Software and Data Integrity Failures
- ✅ Código versionado en Git
- ✅ Validación de inputs en frontend y backend

A09:2021 – Security Logging and Monitoring Failures
- ⚠️ Mejora recomendada: Sistema de logging centralizado
- ⚠️ Mejora recomendada: Alertas automáticas para eventos de seguridad

A10:2021 – Server-Side Request Forgery (SSRF)
- ✅ No aplicable directamente (sin llamadas a URLs externas no confiables)

- Perfil psicológico: 8 variables del cuestionario (motivación, tiempo, etc.)
- Datos de navegación: Logs de acceso (para debugging)

Uso de Datos:
- Gestión académica: Cursos, tareas, calificaciones
- Predicción de riesgo: Análisis con modelo ML
- Comunicación: Foro y comentarios dentro de cursos
- Mejora del servicio: Análisis anónimo de uso

Derechos del Usuario:
- Acceso: Ver todos sus datos personales
- Rectificación: Actualizar información de perfil
- Eliminación: Solicitar eliminación de cuenta y datos
- Portabilidad: Exportar datos en formato estándar

Retención de Datos:
- Datos activos: Durante la vigencia de la cuenta
- Datos históricos: Para análisis y mejoras del modelo ML
- Backup: Según política de backup (ver DRP)

12.4. CONSIDERACIONES ÉTICAS

Uso de Machine Learning:
- Transparencia: Las predicciones incluyen nivel de confianza y explicación de features
- No determinismo: Las predicciones son una herramienta de apoyo, no un juicio definitivo
- Sesgo: Modelo entrenado con datos diversos, variables de género codificadas neutralmente
- Explicabilidad: Features claramente documentadas y visibles en el dashboard

Privacidad de Menores:
- Estudiantes de 14-18 años requieren autorización parental
- Datos académicos accesibles solo a docentes y administradores autorizados
- Perfiles psicológicos protegidos con acceso restringido

Inclusión y No Discriminación:
- Sistema accesible sin discriminación por género, nivel socioeconómico, etc.
- Interfaz diseñada para ser comprensible por todos los usuarios
- Predicciones basadas en datos objetivos, no en características personales protegidas

Responsabilidad:
- Predicciones de riesgo deben usarse para apoyo, no para castigo
- Docentes y administradores deben ser capacitados en uso ético del sistema
- Revisión periódica de predicciones para detectar sesgos o errores

13.1. VINCULACIÓN PROBLEMA → SOLUCIÓN → IMPACTO

El proyecto PAI-Platform aborda directamente el problema de la gestión académica manual y la falta de herramientas proactivas para identificar estudiantes en riesgo académico en el contexto preuniversitario. La solución implementada proporciona:

1. **Gestión Académica Centralizada**: Sistema web completo que automatiza la administración de cursos, tareas y entregas, reduciendo significativamente el tiempo de gestión administrativa y minimizando errores.

2. **Predicción Proactiva de Riesgo**: Modelo de Machine Learning que identifica estudiantes en riesgo desde el primer día basándose en su perfil psicológico y académico, permitiendo intervención temprana antes de que se manifiesten problemas de rendimiento.

3. **Comunicación Integrada**: Sistema de foro por curso que mejora la comunicación entre docentes y estudiantes, facilitando la resolución de dudas y el seguimiento académico.

4. **Visibilidad y Toma de Decisiones**: Dashboards específicos por rol que proporcionan información clara y accionable, permitiendo a docentes y administradores tomar decisiones informadas basadas en datos.
